System Design

how to build systems that scale to millions of users.

what is system design

system design is the process of defining the architecture, components, and data flow of a system to satisfy given requirements. itвҖҷs about making the right tradeoffs between performance, scalability, reliability, and cost.

two types: - high level design (HLD) вҶ’ architecture, components, how they connect - low level design (LLD) вҶ’ classes, databases, APIs, data structures

key concepts before anything else

latency vs throughput

  • latency вҶ’ time to complete one operation (ms)
  • throughput вҶ’ operations per unit time (req/sec)
  • goal: low latency AND high throughput

availability vs consistency

  • availability вҶ’ system is always responding
  • consistency вҶ’ all users see the same data at the same time
  • you often canвҖҷt have both perfectly вҖ” this is the CAP theorem

scalability

  • vertical scaling (scale up) вҶ’ bigger machine, more RAM/CPU
  • horizontal scaling (scale out) вҶ’ more machines

CAP theorem

a distributed system can only guarantee 2 of these 3:

         Consistency
              /\
             /  \
            /    \
           /      \
          /________\
    Availability  Partition
                  Tolerance
  • CA вҶ’ consistent + available (single node, no partitions)
  • CP вҶ’ consistent + partition tolerant (MongoDB, HBase)
  • AP вҶ’ available + partition tolerant (Cassandra, DynamoDB)

in real distributed systems, network partitions WILL happen, so you choose between CP or AP.

ACID vs BASE

ACID (relational databases) - Atomicity вҶ’ transaction is all or nothing - Consistency вҶ’ data is always valid - Isolation вҶ’ transactions donвҖҷt interfere - Durability вҶ’ committed data survives crashes

BASE (NoSQL databases) - Basically Available вҶ’ always responds - Soft state вҶ’ data may change over time - Eventually consistent вҶ’ all nodes will agree eventually

storing data in fast storage (memory) to avoid slow operations (database, network).

cache strategies:

cache aside (lazy loading)

app checks cache
if miss вҶ’ app fetches DB вҶ’ app writes to cache

good for read-heavy workloads

write through

app writes to cache вҶ’ cache writes to DB

data is always consistent, but write latency is higher

write back (write behind)

app writes to cache вҶ’ cache writes to DB asynchronously

fast writes, risk of data loss if cache fails

read through

app reads from cache вҶ’ if miss, cache fetches DB automatically

cache eviction policies: - LRU (Least Recently Used) вҶ’ remove least recently accessed - LFU (Least Frequently Used) вҶ’ remove least accessed overall - TTL вҶ’ expire after time period - FIFO вҶ’ remove oldest entry

where to cache: - browser cache - CDN - reverse proxy (Nginx) - application cache (in memory) - distributed cache (Redis, Memcached)

Redis vs Memcached: - Redis вҶ’ persistent, data structures, pub/sub, more features - Memcached вҶ’ simpler, pure caching, multi-threaded

sharding (partitioning)

split data across multiple databases

users with id 1-1000    вҶ’ shard 1
users with id 1001-2000 вҶ’ shard 2
users with id 2001-3000 вҶ’ shard 3

shard strategies: - range based вҶ’ by value range (easy but hot spots) - hash based вҶ’ hash(key) % num_shards (even distribution) - directory based вҶ’ lookup table for shard location

problems with sharding: - joins across shards are hard - re-sharding is painful - hot shards (one shard gets all traffic)

indexing

speeds up reads by creating a sorted data structure

-- without index: scan every row
SELECT * FROM users WHERE email = 'a@b.com';

-- with index: jump directly to row
CREATE INDEX idx_email ON users(email);
  • B-tree index вҶ’ good for range queries, equality
  • Hash index вҶ’ only equality queries, very fast
  • Composite index вҶ’ multiple columns, order matters
  • indexes slow down writes, speed up reads

message queues

decouple services so they donвҖҷt need to talk directly.

producer вҶ’ [queue] вҶ’ consumer

use cases: - send email after signup (async) - process payment in background - handle traffic spikes (queue absorbs burst) - ensure message delivery even if consumer is down

tools: RabbitMQ, Apache Kafka, AWS SQS, Redis Streams

Kafka vs RabbitMQ: - Kafka вҶ’ high throughput, event streaming, replay messages, log - RabbitMQ вҶ’ traditional queue, complex routing, lower throughput

patterns: - pub/sub вҶ’ one producer, many consumers get the message - point to point вҶ’ one producer, one consumer gets the message - fan out вҶ’ one message goes to multiple queues

rate limiting

prevent abuse and ensure fair usage

algorithms:

token bucket

bucket has N tokens
each request uses 1 token
tokens refill at rate R per second
if no tokens вҶ’ reject request

allows bursts up to bucket size

leaky bucket

requests enter bucket at any rate
bucket leaks at constant rate
if full вҶ’ reject

smooth output, no bursts

fixed window

count requests in current window (e.g. per minute)
if count > limit вҶ’ reject
reset count each window

simple but has edge case at window boundaries

sliding window log

store timestamp of each request
count requests in last 60 seconds

accurate but memory intensive

where to implement: - API gateway - reverse proxy (Nginx) - application code - Redis (distributed rate limiting)

consistent hashing

used in distributed caches and load balancers to minimize re-distribution when nodes are added/removed.

servers and keys are placed on a "ring"

virtual nodes: each server has multiple points on the ring for better distribution.

real-time communication

polling

client asks server every N seconds: "any updates?"
simple but wasteful, high latency

long polling

client asks server, server holds connection open

client immediately makes new request
less wasteful, but still has latency

system design interview framework

when asked to design a system, follow this order:

1. clarify requirements (5 min) - what features are needed? - how many users? (read/write ratio) - what scale? (requests per second, data size) - consistency or availability?

2. estimate scale (5 min) - daily active users - requests per second = DAU Г— requests/day / 86400 - storage = users Г— data per user Г— time - bandwidth = requests/sec Г— data per request

3. high level design (15 min) - draw main components: clients, servers, DB, cache, CDN - explain data flow

4. deep dive (20 min) - focus on most important parts - discuss tradeoffs - handle edge cases

5. wrap up (5 min) - bottlenecks - future improvements - monitoring

example: design URL shortener

requirements: - shorten long URL to short URL (bit.ly/xyz) - redirect short URL to long URL - 100M URLs/day, read heavy (100:1 read to write)

estimation: - writes: 100M/day = 1200/sec - reads: 12000/sec - storage: 100M Г— 500 bytes = 50GB/day

design:

client вҶ’ API gateway вҶ’ app servers вҶ’ cache (Redis)
                                   вҶ’ database (PostgreSQL)

short URL generation: - base62 encoding (a-z, A-Z, 0-9) = 62 chars - 7 characters = 62^7 = 3.5 trillion URLs - hash long URL вҶ’ take first 7 chars - or: auto increment ID вҶ’ base62 encode

database schema:

urls (
  id          bigint primary key,
  short_code  varchar(10) unique,
  long_url    text,
  user_id     bigint,
  created_at  timestamp,
  expires_at  timestamp
)

caching: - cache short_code вҶ’ long_url in Redis - 80/20 rule: 20% of URLs get 80% of traffic - cache those 20% (hot URLs)
